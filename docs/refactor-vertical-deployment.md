# Chuẩn hóa hệ thống quản lý Vertical (Deployment module)

> Tài liệu theo dõi triển khai — cập nhật trạng thái từng phase khi làm xong. Không chỉnh sửa mục "Bối cảnh"/"Kiến trúc" sau khi đã thống nhất, chỉ tick trạng thái ở phần "Task theo phase".

## 1. Bối cảnh & mục tiêu

Cơ chế "vertical" hiện tại (`app/Foundation/Vertical/*`, `Modules/Deployment`, `Modules/Organization`) dùng để THUCHOCVN quản lý các dự án triển khai (tham chiếu: dự án TXNG HTX Tiên Dương, xem `docs/Ke_hoach_trien_khai_TXNG_HTX_Tien_Duong_v1.docx`). Hai vấn đề cần giải quyết:

1. **Danh mục vertical chỉ tạo được bằng seeder code** (`TraceabilityTemplateSeeder`, `UpdateTraceabilityTemplateSeeder`, `NewVerticalsSeeder`, `DeploymentDemoSeeder`) — muốn thêm/sửa vertical phải sửa code + deploy.
2. **Một số chỗ trong engine hardcode theo đúng 1 bộ vertical mặc định** (tên phase `draft`/`surveying`, field-key khảo sát `data_collection_v1`, cột Excel export) — nguy cơ âm thầm gãy tính năng khi có vertical khác.

**Mục tiêu:** bỏ toàn bộ seeder/dữ liệu mặc định cứng, chuyển sang **1 bản mẫu tham khảo tối giản do hệ thống cung cấp sẵn + mỗi tổ chức tự do nhân bản HOẶC tự tạo mới hoàn toàn theo nhu cầu riêng, tuỳ biến độc lập**, quản lý hoàn toàn qua UI thay vì seeder code.

**Trọng tâm ưu tiên:** thiết kế đúng engine tổng quát (nhiều tổ chức, nhiều vertical, có thể nhân bản hoặc không) để THUCHOCVN dùng đi triển khai cho nhiều doanh nghiệp khác nhau. Dự án TXNG HTX Tiên Dương chỉ là ví dụ tham chiếu khi dựng bản mẫu mặc định, **không phải khuôn cứng phải bám sát 100%** — không đưa hạng mục đặc thù (vd. biển đầu bờ/biển khu vực) vào bản mặc định, tổ chức nào cần tự thêm qua UI builder.

### 1.1. Nguyên tắc thiết kế — Organization module là trọng tâm/điểm khởi đầu

Luồng chuẩn của toàn hệ thống, áp dụng cho **mọi tổ chức**, không riêng THUCHOCVN:

1. **Super Admin tạo tổ chức trong module Organization** (`dashboard/organizations`, CRUD đã có sẵn, không đổi) — đây là bước khởi tạo duy nhất và bắt buộc cho bất kỳ tổ chức nào tồn tại trên hệ thống.
2. **Vertical/Deployment là năng lực mở rộng gắn liền vào chính Organization đó**, không phải một hệ thống tách rời hay đặc thù cho 1 org cụ thể. Route `dashboard/organizations/{organization}/verticals` đã đúng nguyên tắc này (thao tác trên resource `Organization` bất kỳ) — kiến trúc mới (mục 2) tiếp tục giữ nguyên tinh thần đó: không có bản ghi, điều kiện, hay đường dẫn nào hardcode theo `organization_id = 3`/THUCHOCVN.
3. **THUCHOCVN chỉ là tổ chức đầu tiên dùng để dựng ví dụ và kiểm thử** (vì có sẵn, có tài liệu TXNG làm tham chiếu) — mọi tổ chức được Super Admin tạo mới sau này, ngay khi vừa tạo xong ở bước 1, phải vào được `verticals` của chính nó và dùng được đầy đủ (nhân bản từ thư viện hoặc tạo mới từ đầu) **mà không cần thêm bất kỳ cấu hình/migrate/seed nào riêng cho tổ chức đó**.
4. **Làm ổn định từng bước theo đúng hướng phụ thuộc**: Organization module (nền tảng, đã ổn định, không đổi) → lớp Vertical core dùng chung toàn hệ thống (Phase 3–6) → gắn tính năng vào trang `verticals` của Organization (Phase 7–8) → kiểm thử lại bằng cách tạo 1 tổ chức **mới hoàn toàn** qua Super Admin để xác nhận tính tổng quát, không chỉ kiểm thử lại trên THUCHOCVN (xem mục 5, bước 6).

Đã xác nhận trạng thái dữ liệu hiện tại (an toàn để xoá seeder, không cần dọn dữ liệu):
- DB (MySQL, `.env` `DB_DATABASE=minhan`) hiện có 0 dòng `vertical_templates`, 0 dòng `organization_verticals`.
- `DeploymentDemoSeeder` chưa từng chạy thật — không có org/user/survey/target demo nào trong DB (org id=2 "Demo Organization" slug `demo` là tổ chức khác, không liên quan seeder này).
- `Modules/Deployment/tests/` hiện rỗng — không có test tự động nào phụ thuộc code sắp xoá/sửa.

## 2. Kiến trúc

### 2.1. Hợp nhất "thư viện mẫu" + "bản của từng tổ chức" vào 1 bảng

Bảng `vertical_templates` thêm cột `organization_id` (nullable):
- `organization_id IS NULL` → **bản mẫu dùng chung** (thư viện chuẩn, System Admin quản lý qua UI mới).
- `organization_id = X` → **bản instance của tổ chức X**, độc lập hoàn toàn sau khi nhân bản/tạo mới (sửa gì không ảnh hưởng bản mẫu hay tổ chức khác).

### 2.2. Thay đổi schema

Quản lý qua JSON manifest của hệ thống (`render_migration_file.json` cho bảng "generated", `render_extension_file.json` cho cột ALTER thêm vào bảng có sẵn) + lệnh `php artisan migration:generate` — **không hand-write migration**.

**Sửa bảng `vertical_templates`** (extension):
| Cột thêm | Kiểu | Ghi chú |
|---|---|---|
| `organization_id` | nullable FK `organizations.id` | null = thư viện, có giá trị = bản của tổ chức |
| `source_template_id` | nullable FK tự tham chiếu `vertical_templates.id` | lưu vết bản mẫu gốc khi nhân bản, null nếu tự tạo từ đầu |
| `status` | string, default `active` | thay vai trò bảng `organization_verticals` |
| `activated_at` | nullable datetime | |
| `activated_by` | nullable FK `users.id` | |

Bỏ ràng buộc unique global trên `code` → unique theo cặp `(organization_id, code)`; validate ở tầng Action khi tạo bản mẫu thư viện (org null) để tránh trùng code giữa các bản mẫu chung (DB không tự chặn NULL trùng nhau).

**Bảng mới `vertical_phases`**: `id, vertical_template_id (FK), key, label, sort_order, is_initial (bool default false), auto_assign_data_collection (bool default false)`.
→ Thay cột JSON `phases`. `is_initial` thay hardcode `'draft'`; `auto_assign_data_collection` thay hardcode `'surveying'`.

**Bảng mới `vertical_checklist_items`**: `id, vertical_phase_id (FK), key, label, is_required (bool default true), sort_order`.
→ Thay cột JSON `default_checklist`.

**Sửa bảng `vertical_config_items`** (extension): đổi khóa từ `(organization_id, vertical_code)` sang `vertical_template_id (FK)`. Giữ nguyên `config_group` (`hierarchy|activity_type|doc_type`), `code`, `label`, `is_active`, `sort_order`.

**Drop bảng `organization_verticals`** — vai trò hấp thụ vào cột mới trên `vertical_templates`.

**Giữ JSON** cho `default_roles` (danh sách suffix vai trò — sửa qua form thêm/xoá dòng đơn giản, không cần bảng riêng) và `sidebar_config`/`export_config` (ít thay đổi, không phải trọng tâm lần này).

Dọn cột chết: `phases`, `default_checklist`, `default_activity_types`, `default_legal_doc_types`, `default_hierarchy` trên `vertical_templates` (chuyển sang bảng con), cột `export_adapter` (đã dead — không có trong `$fillable`).

### 2.3. Sửa lỗi hardcode trong engine

| File | Hiện tại | Sửa thành |
|---|---|---|
| `app/Foundation/VerticalDefinition.php` | thiếu `dataCollectionTemplateSlug()` | thêm `dataCollectionTemplateSlug(): ?string`, `initialPhaseKey(): string`, `autoAssignsDataCollection(string $phaseKey): bool` |
| `app/Foundation/Vertical/DatabaseVertical.php` | đọc cột JSON | implement từ quan hệ `phases`/`checklistItems`; `phases()`/`defaultChecklist()` build lại đúng shape cũ để **không phải sửa shape dữ liệu ở nơi tiêu thụ** |
| `app/Foundation/Vertical/VerticalTemplate.php` | không có quan hệ | thêm `phases()` (HasMany, order `sort_order`, eager `checklistItems`), `configItems()`, `sourceTemplate()`, `clones()` |
| `app/Foundation/VerticalRegistry.php` | `resolve(string $code)` toàn cục | `resolveForOrganization(int $organizationId, string $code)`; thêm `libraryTemplates()`; cache key theo `{organizationId}:{code}` |
| `Modules/Deployment/app/Actions/CreateDeploymentTargetAction.php` | `'current_phase' => 'draft'` | `'current_phase' => $vertical->initialPhaseKey()` |
| `Modules/Deployment/app/Actions/AdvancePhaseAction.php` | `if ($nextPhase === 'surveying')` | `if ($vertical->autoAssignsDataCollection($nextPhase))` |
| `Modules/Deployment/app/Actions/AssignDataCollectionSurveyAction.php` | `method_exists()` luôn false → hardcode `data_collection_v1` | `$vertical?->dataCollectionTemplateSlug() ?? 'data_collection_v1'` |

**Giới hạn chấp nhận (không sửa lần này):** `ChecklistAutoTickAction`, `SyncOrgFromSurveyAction`, `DeploymentExportController::template()` vẫn hardcode field-key theo schema khảo sát `data_collection_v1` và 2 loại cột Excel (`donvi`, `nhatky`). Tổng quát hoá field-key theo từng vertical tuỳ biến là hạng mục lớn riêng, để lại fast-follow. Vertical mặc định vẫn dùng đúng `data_collection_v1` nên hoạt động bình thường; tổ chức tự đổi khảo sát khác thì 3 tính năng này lặng lẽ không kích hoạt (không crash).

### 2.4. Điểm chạm cần cập nhật đồng bộ (impact list — đã grep xác nhận)

`VerticalRegistry::resolve()` được gọi tại:
- `app/Foundation/Vertical/ActivateVerticalAction.php` (sẽ xoá, thay bằng Clone/CreateFromScratch action)
- `app/Http/Middleware/RequireVertical.php`
- `app/View/Composers/SidebarComposer.php` (đang `->map(fn ($ov) => VerticalRegistry::resolve($ov->vertical_code))` trên danh sách `OrganizationVertical` — phải đổi sang query trực tiếp `vertical_templates` theo `organization_id`)
- `Modules/Organization/app/Http/Controllers/OrganizationVerticalController.php` (3 chỗ)
- `Modules/Deployment/app/Http/Controllers/DeploymentLandingController.php` (2 chỗ, cùng pattern `->map` như SidebarComposer)

`OrganizationVertical` model được dùng tại 8 file (middleware, actions, composer, controller) — tất cả sẽ đổi sang thao tác trực tiếp trên `VerticalTemplate` (dòng có `organization_id`).

`$request->attributes->get('_vertical')` được **~13 controller Deployment tiêu thụ** (Checklist, DataCollection, Target, Project, Dashboard, Progress, Issue, Readiness, Report, Export, Media) — **không cần sửa các file này**, vì chúng chỉ tiêu thụ object `VerticalDefinition` được `RequireVertical` middleware set sẵn; chỉ cần middleware resolve đúng là toàn bộ downstream tự động đúng theo interface hiện có. Đây là điểm giảm rủi ro quan trọng — diện sửa code hẹp hơn nhiều so với số nơi "liên quan" tới vertical.

## 3. Rủi ro & biện pháp đảm bảo ổn định

| Rủi ro | Biện pháp |
|---|---|
| Đổi schema `vertical_templates`/`vertical_config_items` khi đang có dữ liệu thật | Đã xác nhận DB đang có 0 dòng ở các bảng liên quan → an toàn tuyệt đối, không cần script di trú dữ liệu cũ. Nếu sau này phát hiện dữ liệu phát sinh trước khi merge, phải viết script chuyển `default_checklist`/`phases` JSON cũ sang bảng con trước khi drop cột. |
| `VerticalTemplate::booted()` chỉ invalidate cache khi ghi qua Eloquent | Toàn bộ action mới (Clone/CreateFromScratch/builder CRUD) phải dùng Eloquent, không dùng raw `DB::table()->insert()`, để cache `VerticalRegistry` không bị stale. |
| Cache TTL 1 giờ (`Cache::remember(...now()->addHour())`) có thể trả dữ liệu cũ ngay sau khi sửa qua UI builder | Gọi `VerticalRegistry::clearCache()` tường minh sau mỗi thao tác builder (thêm/sửa/xoá/reorder phase, checklist, config item), không chỉ dựa vào model event của `VerticalTemplate` (vì phases/checklist là model riêng, không tự trigger `booted()` của `VerticalTemplate`). |
| Spatie role động `{code}_{suffix}` tạo trùng tên giữa các tổ chức | Đã xác nhận `config/permission.php` bật `'teams' => true` với `team_foreign_key = organization_id`, và middleware `SetCurrentOrganization` đã gọi `setPermissionsTeamId()` theo request → role tách theo team tự động, không cần code thêm gì. |
| `RequireVertical` đổi cách resolve có thể làm sập toàn bộ route `dashboard/{vertical}/...` nếu sai | Triển khai theo thứ tự: sửa xong toàn bộ lớp core (2.3) + kiểm thử bằng tinker/route:list trước khi đụng tới UI; giữ nguyên contract `VerticalDefinition` (không đổi tên method cũ, chỉ thêm method mới) để 13 controller tiêu thụ `_vertical` không cần sửa. |
| Xoá bảng `organization_verticals`/model `OrganizationVertical` khi vẫn còn 8 file tham chiếu | Liệt kê đủ ở mục 2.4 — sửa tuần tự theo dependency: core trước (registry/model/interface) → action (activate/deactivate cũ) → middleware/composer/controller. Chạy `grep -rn "OrganizationVertical"` lại lần cuối trước khi xoá file để đảm bảo không sót. |
| Không có test tự động cho Deployment module | Bổ sung kiểm thử thủ công chi tiết ở mục 5 làm nghiệm thu bắt buộc trước khi coi hoàn thành; cân nhắc viết Feature test tối thiểu cho `AdvancePhaseAction`/`CreateDeploymentTargetAction` sau khi refactor xong nếu còn thời gian (không bắt buộc trong scope này). |
| Đổi schema production (không phải chỉ local) | `php artisan migration:generate` (không `--fresh`) rồi `php artisan migrate` thông thường trên môi trường có dữ liệu — tuyệt đối không dùng `--fresh`/`migrate:fresh` ngoài local vì sẽ xoá sạch DB. |
